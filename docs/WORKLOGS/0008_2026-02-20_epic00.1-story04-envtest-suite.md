## Objective

Implement STORY_04: create the envtest bootstrap infrastructure so that future integration tests can run against a real Kubernetes API server without a live cluster. No reconcile logic — just the test suite plumbing.

## Work Completed

### 1. CRD YAML files

- Created `testdata/crds/result_crd.yaml` — minimal hand-written CRD for `core.k8sgpt.ai/v1alpha1 Result`, derived from `api/v1alpha1/result_types.go`. Includes all ResultSpec fields (backend, kind, name, error array with text/sensitive, details, parentObject) and an empty status with `x-kubernetes-preserve-unknown-fields: true`.
- Created `testdata/crds/remediationjob_crd.yaml` — full CRD copied from DEPLOY_LLD.md §2.1, including CEL `x-kubernetes-validations` immutability rules on fingerprint, sourceType, and sinkType.

### 2. envtest suite files

- Created `internal/provider/k8sgpt/suite_test.go` (package `k8sgpt_test`) — TestMain starts `envtest.Environment` pointing at `../../testdata/crds`, registers both schemes via `v1alpha1.NewScheme()`, creates a `client.Client`, runs tests, and stops envtest. Exports `k8sClient` as a package-level var for integration tests. Includes `TestSuite_StartsAndStops` smoke test that lists `RemediationJobList`.
- Created `internal/controller/suite_test.go` (package `controller_test`) — identical pattern for the controller package.
- Both files include the graceful skip pattern: if `KUBEBUILDER_ASSETS` is unset and `setup-envtest` is unavailable, exit 0 (skip, not fail).

### 3. go.mod update

- Promoted `sigs.k8s.io/controller-runtime v0.19.3` from indirect to direct dependency in `go.mod`.
- Ran `go mod tidy` to update go.sum.

---

## Key Decisions

- Used `v1alpha1.NewScheme()` (already defined in `remediationjob_types.go`) instead of building the scheme manually in each suite. This registers both Result and RemediationJob groups and avoids duplication.
- Used `package k8sgpt_test` and `package controller_test` (external test packages) for suite files as required. Internal test package would conflict since unit test files in those packages already use the internal package name.
- CRD path is `../../testdata/crds` (two levels up from each package directory to the repo root). Both packages are two levels deep from root (`internal/provider/k8sgpt/` and `internal/controller/`).
- The `testdata/crds/` directory is at the repo root (not inside `internal/`) so both suite files can share the same CRD set via the same relative path.
