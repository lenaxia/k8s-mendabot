### 3. OpenCode CLI Configuration Format — ROOT CAUSE IDENTIFIED

**Problem**: All attempted formats were rejected by opencode CLI.

**Tried formats (all rejected)**:
1. `{"model":"glm-4.7","providers":{"openai":{"apiKey":"sk-xxx","baseURL":"https://..."}}` — Error: `Unrecognized key: "providers"`
2. `{"model":"glm-4.7","api_key":"sk-xxx","api_base":"https://..."}` — Error: `Unrecognized keys: "api_key", "api_base"`
3. `{"model":"glm-4.7","apiKey":"sk-xxx","baseURL":"https://..."}` — Error: `Unrecognized keys: "apiKey", "baseURL"`
4. `{"model":"openai/glm-4.7"}` — Error: `Model not found: openai/glm-4.7`
5. `{"model":"glm-4.7","provider":{"openai":{"model":"glm-4.7","options":{"baseURL":"...","apiKey":"..."}}}}` — Error: `Unrecognized keys: "apiKey", "baseURL"` (keys should be in options)
6. `{"provider":{"openai":{"model":"glm-4.7","options":{"baseURL":"...","apiKey":"..."}}}}` — Error: `Config file is not valid JSON(C): CloseBraceExpected`
7. Same as 6 with different quoting — Error: `CloseBraceExpected`

**Root cause**: Two problems compounded:

1. **Wrong key `"providers"` (plural) in NOTES.txt** — the correct opencode config key is `"provider"` (singular). `NOTES.txt` had the wrong example. This was the cause of all "Unrecognized key: providers" errors. **Fixed in this worklog: `charts/mendabot/templates/NOTES.txt` corrected.**

2. **Built-in `"openai"` provider is not the right provider for a custom baseURL.** The built-in `openai` provider in opencode is hard-wired to `api.openai.com`. Using `options.baseURL` inside it may work for some versions, but the canonical way for a custom OpenAI-compatible endpoint (like `https://ai.thekao.cloud/v1`) is to define a **custom provider** using `"npm": "@ai-sdk/openai-compatible"`. This is the same pattern opencode docs use for Ollama, LM Studio, llama.cpp, etc.

**Correct format for `provider-config` (for GLM-4.7 on thekao.cloud):**

```json
{
  "provider": {
    "thekao": {
      "npm": "@ai-sdk/openai-compatible",
      "name": "domain.tld",
      "options": {
        "baseURL": "https://ai.domain.tld/v1",
        "apiKey": "sk-..."
      },
      "models": {
        "glm-4.7": {
          "name": "GLM-4.7"
        }
      }
    }
  },
  "model": "openai/glm-4.7"
}
```

The `OPENCODE_CONFIG_CONTENT` env var (which `entrypoint-opencode.sh` sets from `AGENT_PROVIDER_CONFIG`) is loaded by opencode as **inline config** — the highest precedence config layer per `https://opencode.ai/docs/config`. The format must conform to `https://opencode.ai/config.json` schema.

**Status**: Root cause identified. Secret must be recreated with the correct format above.

### 4. LLM Readiness Check False Negative — ROOT CAUSE IDENTIFIED

**Problem**: With `watcher.llmProvider=openai`, the readiness checker failed:
```
llm/openai: secret "llm-credentials-opencode" not found in namespace "default"
```
Even though the secret existed in namespace `default`.

**Root cause**: The `OpenAIChecker` reads the secret via the controller-runtime **informer cache**, not a direct API call. The informer cache for `corev1.Secret` is scoped to `cfg.AgentNamespace` (set via `AGENT_NAMESPACE` env var). The cache is populated asynchronously on manager start. If the readiness check runs before the informer has fully synced (which can happen during the first few reconcile loops immediately after pod start), the secret appears absent even though it exists in the API server.

This is a known controller-runtime pattern: the informer cache may not be warm at the first reconcile. The `CachedChecker` TTL caches the result of the *first* check, so if the first check fires before cache sync, it will return "not found" and cache that failure for the full TTL (30 seconds by default via `provider.ReadinessCacheTTL`). After TTL expiry the next check will use the warmed cache and succeed.

**Resolution already in place**: Setting `watcher.llmProvider=""` disables the gate. This is acceptable for the current debugging phase but means agents will attempt to launch even if the secret is absent. Once the correct secret format is confirmed working, re-enable with `watcher.llmProvider=openai`.

**No code fix required** — this is a timing artifact. The operator should wait ~30s after secret creation before expecting the first successful readiness check.

### 5. Watcher Pod "v1: Unauthorized" on Restart

**Problem**: New watcher pods fail on start:
```
unable to start manager: failed to determine if *v1.Secret is namespaced:
failed to get API group resources: unable to retrieve the complete list of
server APIs: v1: Unauthorized
```
The old pod (running for 17m) continued to work.

**Root cause**: Unknown. The most likely cause is that the watcher's ServiceAccount token was rotated or invalidated when Helm re-applied the ServiceAccount manifest (which can happen if the SA annotation changed). A secondary possibility is a brief apiserver hiccup during the Helm upgrade. The old pod holds a valid token from before the upgrade; the new pod gets a token that briefly fails discovery.

**Workaround**: Old pod is still running and processing findings. Delete the new failed pod; it will not reschedule (Deployment restart policy). Force a clean restart:
```bash
kubectl rollout restart deployment/mendabot -n default
```

**Investigation needed**: Check `kubectl describe serviceaccount mendabot-watcher -n default` and `kubectl get secrets -n default | grep mendabot` to confirm the SA token secret is intact and not expired.

- **Custom provider pattern**: GLM-4.7 on thekao.cloud must be configured as a custom OpenAI-compatible provider (`"npm": "@ai-sdk/openai-compatible"`), not via the built-in `"openai"` provider entry. The built-in `openai` is for `api.openai.com`.
- **NOTES.txt fixed**: Example `provider-config` in `charts/mendabot/templates/NOTES.txt` corrected from `"providers"` (wrong) to `"provider"` (correct), and updated to use the custom provider pattern.
- **LLM readiness check**: Leave `watcher.llmProvider=openai` enabled in production. The false negative is a startup timing artifact, not a bug. Cache TTL ensures recovery within 30s.
