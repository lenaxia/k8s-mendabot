## Work Completed

### 1. k8sgpt dependency evaluation

Read the k8sgpt source (`~/personal/k8sgpt`). All analysers (Pod, Deployment, PVC, Node,
StatefulSet, Job, etc.) are ~50–150 lines of straightforward Kubernetes API field checks.
None of the logic requires an external operator. The dependency costs more than it gives:
mandatory deployment prerequisite, redundant binary in the agent image, added latency from
k8sgpt's polling interval.

### 2. Architecture design

Evaluated three options for the provider structure:
- Option A: one monolithic `NativeProvider` handling all kinds — rejected (violates SRP,
  single `ObjectType()` can only return one type)
- Option B: one provider per kind, each returning a different `ProviderName()` — rejected
  (breaks `RemediationJob.spec.sourceType` consistency)
- **Option C selected**: one `SourceProvider` per kind, all returning `ProviderName() =
  "native"`. Each registered separately in `main.go`. The existing provider registration
  loop handles this without change.

### 3. Interface change: Fingerprint moves to domain

`SourceProvider.Fingerprint(f *Finding)` is domain logic, not provider logic. Every
provider implements the same algorithm. The correct fix is a package-level function
`domain.FindingFingerprint`. This eliminates the `fingerprintFor()` function in
`internal/provider/k8sgpt/reconciler.go` and the `TestFingerprintEquivalence` test that
only existed to detect divergence between two copies of the same algorithm.

### 4. Analysers in scope

Four core analysers for v1:
- `PodProvider`: CrashLoopBackOff, ImagePullBackOff, OOMKilled, ErrImagePull,
  CreateContainerConfigError and similar, non-zero exit code, unschedulable pending pods
- `DeploymentProvider`: `spec.replicas != status.readyReplicas` with scaling transient
  exclusion; `Available=False` condition check with Reason/Message in error text
- `PVCProvider`: `Phase == Pending` with `ProvisioningFailed` event
- `NodeProvider`: `NodeReady == False/Unknown`, non-standard conditions present

Deliberately excluded: `LogAnalyzer` (noise), `ConfigMapAnalyzer` (not actionable),
Ingress/Service/Gateway/KEDA/Kyverno (edge cases with unclear remediation paths).

Identified for STORY_10 and STORY_11 (added in follow-up session):
- `StatefulSetProvider`: ready replicas mismatch + `Available=False` condition
- `JobProvider`: precise `failed > 0 AND active == 0 AND completionTime == nil` detection

### 5. Meaningful improvements over k8sgpt

1. Signal fidelity — more precise detection conditions (e.g. Job active-failure check)
2. Richer error context — include condition `Reason` and `Message` strings, not just
   replica counts
3. Stabilisation window — configurable observation period before a finding becomes a
   `RemediationJob`; prevents noise from transient restarts

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| One provider per kind (Option C) | SRP, consistent `ProviderName()`, no change to registration loop |
| `domain.FindingFingerprint` standalone function | Eliminates algorithm duplication and the equivalence test |
| `ProviderName() = "native"` for all native providers | Consistent `sourceType` in `RemediationJob` |
| Exclude `LogAnalyzer` | noise-to-signal ratio too high for automated remediation |
| Exclude readiness probe event detection from PodProvider | Event-order-dependent, noisy; removed in follow-up |
| Stabilisation window: Option C (in-memory map) | No new CRD; restart resets clock but is not a correctness failure; acceptable for default 2-min windows |
