## Work Completed

### 1. BUG-01 — Double-counting RecordSelfRemediationAttempt
- **Root cause:** `internal/provider/provider.go` called `RecordSelfRemediationAttempt(..., true)` unconditionally at RemediationJob creation dispatch time, then `internal/controller/remediationjob_controller.go` called it again with the real outcome at completion. Every self-remediation logged 2 attempts — one pre-counted as success.
- **Fix:** Removed the premature call in `provider.go`. The controller already correctly records it once at completion.
- **Test:** `TestReconcile_SelfRemediation_NoDoubleCountAttempt` in `internal/provider/provider_test.go` — verifies zero metric increment at dispatch for a self-remediation finding.

### 2. BUG-02 — RecordMaxDepthExceeded metric dead at default config
- **Root cause:** `internal/provider/provider.go` had `if finding.ChainDepth > 3 { metrics.RecordMaxDepthExceeded(...) }` hardcoded. With `SelfRemediationMaxDepth=2` (default), `job.go` drops findings at depth > 2 before they reach the provider, so no finding with depth > 3 ever arrives. The metric was permanently dead.
- **Fix:** Changed threshold to `if finding.ChainDepth >= r.Cfg.SelfRemediationMaxDepth`. Also adjusted the deep-cascade log from `> 2` to `> 1` so it fires for any actual cascade (depth >= 2).
- **Test:** `TestReconcile_MaxDepthExceeded_MetricFires` in `internal/provider/provider_test.go` — verifies metric increments when `ChainDepth >= SelfRemediationMaxDepth`.

### 3. BUG-03/04 — Race detector failures in concurrent tests
- **Root cause:** `TestJobProvider_ConcurrentReconciliationRace` (`internal/provider/native/job_test.go`) and the concurrent test in `internal/cascade_prevention_integration_test.go` both passed a single shared `parentRJob` pointer into `fake.NewClientBuilder().WithObjects(parentRJob)` from multiple goroutines. The fake client builder mutates `ResourceVersion` on the passed object during `Build()`, causing a concurrent write race.
- **Fix:** Deep-copied `parentRJob` inside each goroutine via `parentRJob.DeepCopyObject().(client.Object)` before passing to `WithObjects`. Applied to both files.

### 4. BUG-05 — Lazy init race for cascadeChecker and circuitBreaker
- **Root cause:** Both components were initialized inside `Reconcile()` with a nil check but no synchronization. Two concurrent `Reconcile()` calls could both observe nil and write the pointer simultaneously. The existing `initOnce` pattern for `firstSeen` was not applied to these two fields.
- **Fix:** Added `initCascadeOnce sync.Once` and `initCBOnce sync.Once` to `SourceProviderReconciler`. Replaced the nil-check pattern with `sync.Once.Do(...)` for both components, consistent with the existing `initFirstSeen()` pattern.

### 5. BUG-06 — Ghost targetRepoOverride in CRD YAML
- **Root cause:** STORY_04 (upstream routing) was intentionally removed during design, but `targetRepoOverride` was not removed from the CRD YAML. The field had no corresponding Go struct field, so Kubernetes would accept it but the controller would silently ignore it.
- **Fix:** Removed the 3-line `targetRepoOverride` block from `deploy/kustomize/crd-remediationjob.yaml` and from `charts/mendabot/crds/remediationjob.yaml`.

### 6. BUG-07 — Dead code isValidGitHubIdentifier()
- **Root cause:** `internal/config/config.go` defined `isValidGitHubIdentifier()` but it was never called anywhere. It was a remnant of removed STORY_04 upstream repo validation.
- **Fix:** Deleted the function entirely.

### 7. Missing provider-level test for DISABLE_CASCADE_CHECK
- **Added:** `TestReconcile_DisableCascadeCheck_Bypasses` in `internal/provider/provider_test.go` — verifies that with `DisableCascadeCheck: true`, a self-remediation finding results in a RemediationJob creation and `cascade_suppressions_total{suppression_type="infrastructure_cascade"}` stays at zero.

### 8. Added metric accessor functions for test inspection
- **Added:** `SelfRemediationAttemptsTotal()`, `MaxDepthExceededTotal()`, `CascadeSuppressionsTotal()` exported accessors in `internal/metrics/metrics.go` — allows `testutil.ToFloat64()` in tests without exposing the full Prometheus collector.

### 9. Updated story DoD status
- **STORY_06:** Updated from "Not Started" → "In Progress". Checked boxes for all implemented metric work. Left unchecked: Kubernetes Events, Grafana dashboard, alert integration.
- **STORY_07:** Updated from "Not Started" → "Complete". All config env vars implemented, validated, and wired. Added note that inline validation (not a named `validateCascadeConfig()` function) is the actual implementation.

## Key Decisions

- **BUG-01 fix strategy:** Remove the early-dispatch call entirely rather than add a flag. The controller's completion-time recording is the correct and only place for attempt tracking. Adding a "dispatched" vs "completed" split would require metric label changes and is unnecessary complexity.
- **BUG-02 threshold:** `>= SelfRemediationMaxDepth` (not `> SelfRemediationMaxDepth`) — at exactly max depth, the finding has already been allowed through (job.go enforces `> MaxDepth`), so the metric should fire to signal the cascade is at its configured ceiling.
- **sync.Once for cascade/CB init:** Applied the same pattern already used for `firstSeen`. The `initCBOnce` remains inside the `if finding.IsSelfRemediation` block so non-self-remediation reconciles never initialize it. This is correct — circuit breaker is only needed for self-remediations.
- **STORY_06/07 status:** Not marking STORY_06 Complete because Kubernetes Events, Grafana dashboard, and alert rule integration are genuinely absent. Marking STORY_07 Complete because all 5 env vars are implemented and tested — the design doc named a function that was never needed.
